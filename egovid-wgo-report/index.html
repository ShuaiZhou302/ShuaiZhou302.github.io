导读：为什么要给第一视角人手视频做子任务标注

Egocentric human data（第一视角人手数据）用头戴或胸挂相机记录人如何操作物体：画面晃、手挡、动作密，和桌面第三人称机器人演示很不一样。要把长程演示变成可学习的监督信号，通常需要切成可执行的原子子任务，并写成短指令。

子任务边界与文案正在成为 VLA / 链式推理 / 奖励建模等管线的核心输入。人工逐小时标注无法跟上数据规模，因此需要可复现的自动标注管线。

已有公开工作（地图）

  • Egocentric 数据族：EgoVerse / EgoDex / EgoLive 等第一人称人手操作演示，面向规模化 human 数据。
  • HomER：WGO-Bench 里最难的 egocentric 子集；本报告消融固定 HomER 25 / 470 gold 段。
  • Vitra：偏「先用手部/运动信号切，再 caption」的启发；EgoVid Annotate 生产线用 HaWoR 腕速实现切段，而非直接跑 Vitra 模型。
  • WGO-Bench / Macrodata blog：公开「切段 + 短句」评测协议与 Gemini 参考分，以及约 $2.64/视频小时（batch）的端到端开销。
  • Scale Labs dense video captioning：在已切好的 clip 上做稠密描述；不做 raw episode 的切段问题。

本页报告 EgoVid Annotate 在 HomER 上、对标 WGO 协议的可复现消融：什么涨分、什么失败、成本如何与公开数字对照。名词解释见后文背景章。

评测范围：本页分数与推荐配置均锚定 HomER。WGO-Bench 中的 robot 视角子集，以及其他 egocentric 库上的同协议结果尚未在本页报告，留待后续扩展。

1. 背景：这些名字分别是什么

Macrodata 公开报告在谈什么

Macrodata · Annotating Robot Video Subtasks 把「长视频 → 可执行子任务 + 短句」做成公开问题：发布数据集 WGO-Bench、三种分数（分段 / 固定边界标注 / 端到端），以及 Gemini 等参考分。 本报告在同一评测协议上，给出 EgoVid Annotate 的可复现消融：涨分项、失败配置与推荐路径。

Egocentric 数据与 WGO-Bench

规模化第一人称人手数据常见族系包括 EgoVerse / EgoDex / EgoLive 等；它们提供长程 human 演示，但「如何切成可执行子任务 + 短指令」仍需标注协议与评测。 WGO-Bench 把这一问题做成公开基准（约 100 个 episode），其中 HomER 是第一视角人类操作子集——画面晃、手挡、动作密，通常比桌面第三人称更难。 本报告固定评测：HomER 25 集 / 470 个 gold 子任务段。 Macrodata 全量 100 集 Segment F1≈0.306 是 Gemini 主线 headline；HomER-only 更公平的对照约 0.227。 不宜与全量 100 集 headline（0.306）直接对比。同协议下其他子集的结果见导读范围声明。

EgoVid Annotate：生产默认 vs 本报告实验线

EgoVid Annotate 是我们的 egocentric 人手视频自动标注系统。它有两条需要分清的路径：

  • 生产默认管线（日常批标）：HaWoR 手重建 → 腕速平滑与 minima 切段 → 段内 raw 抽帧写短句 → merge judge / rewrite。 细节见 附录 D。启发来自 Vitra 一类「先运动再 caption」工作流，但切段后端是 HaWoR 腕速，不是 Vitra 模型。
  • 本报告 WGO 实验线（对标 Macrodata）：contact sheet + Qwen 视觉语言模型切段 → 锁边界后标注 / 多候选 selector。 正文消融主要讲这条线。

模型栈以 Qwen 为主：分段用 Qwen3.6-27B（快、便宜、可复现）， 标注 / judge / selector 用 Qwen3.5-397B（句子质量与选优）。 相对 Gemini 参考栈，同一套权重与 prompt 可在不同机器上复跑，消融结论可核对。

GEPA 是什么（定义）

  1. GEPA 本身：Macrodata 用来自动搜索更好 prompt 的方法/过程。我们写这篇报告时没有在推理阶段重跑 GEPA。
  2. 我们实际用的东西：搜索结果沉淀下来的一段英文分段 prompt（他们内部名 completed_events_duration_prior_v1）。
  3. 实质：就是在发给视觉语言模型的 text prompt 里加入一组硬规则 ——只切「已完成的操作事件」、多数段约 2–10 秒、不要把靠近/微调/收回拆成假事件、不要把不同的 pick/place 乱合并。 不是新模型，也不是后处理脚本。
  4. 怎么用：整集 contact sheet 图像 + 这段英文规则 → 一次粗分。样例 Step 02 折叠区为英文全文；概念与实现见 附录 B，原文下载见 附录 F。

后面会反复出现的几个词

  • 时间窗口：视频时间轴上的一段连续区间（例如 84–94 秒），不是软件 UI 窗口。
  • 第一遍加密切（S1）：提高切段密度以抬召回（容易切碎）。详见 附录 B。
  • 第二遍局部精修（S2）：在粗边界附近的一小段时间里再切一次。
  • 窗口不外扩（pad=0）：精修时只看粗边界这段，不向两侧多「偷看」几秒。
  • 盖住完整动作（full-cover）：这段里看得见的完成事件都要切到，别切成摆弄碎片。
  • selector / judge：selector 从多条候选标签里选一句定稿；judge 只用于打分，判断预测句与 gold 是否同一完成事件。
  • HaWoR：手部重建，用来得到真手腕轨迹再做 hand-crop（贵；默认标注仍可用 raw 帧）。
  • Qwen3.6-27B：分段主模型（日志有时写作 28B endpoint，同一服务）；Qwen3.5-397B：标注 / judge / selector。
易混淆的视觉输入： contact sheet（分段用时间戳拼图）≠ 整帧 temporal collage ≠ L1 邻段 sheet ≠ L2 YOLO/proxy hand-collage ≠ HaWoR true hand-crop。

2. 怎么打分（三句话 + 一个玩具例子）

完整公式与直觉见 附录 A。正文只记定义与玩具例。

  1. Segment F1：只评时间切分（刀工）；IoU≥0.75 一对一配对。
  2. 固定边界 Label Acc：时间用 gold，只评句子是否同一完成事件（文案）。
  3. Semantic E2E F1：先时间配对，再对配对成功的 pair 做语义 judge；刀工与文案都过才算。

F1 越大越好，上限 1.0。玩具例子（与 scorer 一致）：

Gold：G0[0,3], G1[3,6], G2[6,10] · Pred：P0[0.5,2.8], P1[2.8,5.5], P2[5.5,8], P3[8,9.5]

  1. Outer snap：P0.start→0，P3.end→10
  2. IoU：P0–G0≈0.933，P1–G1≈0.781
  3. n_match=2 → P=0.50，R≈0.67，F1≈0.571

3. Contact sheet：模型实际在「看」什么

我们不把整段 MP4 作为视觉输入直接提交给模型，而是每隔 0.5 秒抽一帧，缩到约 224×144， 每张 sheet 20 格（5×4），格子上画黄色时间戳。 一张 sheet 大约覆盖 10 秒；一集长视频会拆成多张 sheet。

6. 消融故事：我们怎样走到 0.1517

按时间线叙述；主表与图跟在后面。实验细节（做法卡、pad 消融）默认折叠。

6.1 分段：从假切点到「盖住完整动作」

  1. 原 EgoVid Annotate rule-based 腕速切段：过碎（F1≈0.095）。
  2. Contact sheet 分片（max_sheets=3，每次最多送 3 张）：外形接近公开报告做法，但接缝处造假边界。
  3. 整集一次 + legacy prompt（尚无切段规则清单）：假切点没了，却欠分割。
  4. 换成 切段规则清单（GEPA）：到 0.137，仍偏少。
  5. 第一遍加密切（S1）：召回上升，但过分割。
  6. 第二遍局部精修：方向对；最终 窗口不外扩 + 盖住完整动作（S2 · pad=0 · full-cover）到 0.2031。 （另测过用算法补覆盖的后处理：不如把「盖住完整动作」写进 prompt。）
  7. 相邻 merge 三种规则后处理（不是再调模型）：时间轴更好看，但 F1 全掉——不要默认 merge。

Segment F1（主路径）

8. 附录：概念、公式、实现与成本记账

正文讲「试了什么、分数怎么变」；本附录讲实现细节（公式/表可能中英混排）。 目录： A 得分 · B 分段概念 · C 标注/E2E 术语 · D 生产管线 · E 视觉输入 · F Prompt · G 成本记账 · H 效度。

A. 得分方法：直觉 + 公式

A.1 Segment F1（只评刀工）

直觉：不管句子写得好不好，只看时间段有没有切对。像用剪刀剪录像：剪对了就算。

  1. Outer snap：每个 episode 的第一条预测起点、最后一条预测终点对齐到视频起止。
  2. 若 pred 与 gold 的时间 IoU ≥ 0.75，可一对一 greedy 配对（一个 pred 不能配多个 gold）。
  3. 全集 micro：
P = m / n_pred
R = m / n_gold
F1 = 2·P·R / (P+R)
其中 m = temporal matches，本子集 n_gold = 470

玩具例（§2）：m=2，n_pred=4，n_gold=3 → P=0.50，R≈0.667，F1≈0.571。

A.2 固定边界 Label Acc（只评文案）

直觉：时间边界用 gold「作弊」给定，只考模型会不会把这段动作写成对的短句。

Acc = n_semantically_correct / n_gold
（由 LLM judge 判定 predicted label 与 gold 是否同一完成事件）

A.3 Semantic E2E F1（刀工 + 文案）

直觉：先要时间配对成功，再对配对成功的 pair 做语义 judge；两边都过才进分子。

先按 A.1 得到 temporal matches
再对每个 match pair 做语义 judge → m_sem
P_e2e = m_sem / n_pred
R_e2e = m_sem / n_gold
F1_e2e = 2·P_e2e·R_e2e / (P_e2e+R_e2e)

因此：分段 F1=0.2031 时，即使标签全对，E2E 也会被时间匹配 ceiling 卡住。邻段重标掉到 0.108 时，IoU match 仍是 79，掉的是语义 match。

B. 分段概念卡

欠分割 vs 第一遍加密切（S1）

直觉：欠分割=切太少（漏动作）；第一遍加密切=故意催模型多切，抬召回，但容易切碎。

输入/输出：整集 contact sheet + 偏短段的 duration prior → 更多、更短的时间段。

实现：改分段 prompt 的时长先验 / 加密切指令；全量 25 集。消融表行：第一遍加密切（S1）（F1 0.1556，pred 558）。

第二遍局部精修早期版（S2，尚未盖住完整动作）

直觉：粗分之后，在每条粗边界附近开局部时间窗，再看一张「短片联系表」细切一次。

输入/输出：局部时间窗口 contact sheet（同 0.5s / 5×4 layout）→ 精修后的边界列表。

实现：早期常用窗外扩≈1 秒（pad≈1），prompt 尚未要求盖住完整动作。消融表行：第二遍局部精修早期版（F1 0.1674）。

窗口不外扩（pad=0）

直觉：精修窗口不要向外「多看几秒」，避免把邻段动作带进当前窗。

实现：局部窗 = 粗边界区间，外扩秒数=0。窗口外扩消融见折叠表（0.5/1/2）。

盖住完整动作（full-cover local prompt）

直觉：告诉模型：这个短窗口里看得见的完成事件都要盖住，但不要切成「调整/摆弄」碎片。

输入/输出：窗口不外扩的局部 sheet + full-cover 英文规则 → 本报告最强分段 0.2031。

实现：prompt 文件 s2_fullcover_refine.md（样例 Step 03 折叠区）。对比：用算法做 midpoint 补覆盖 后处理 只有 0.1635——把约束写进 prompt 更有效。消融表行：局部再切：窗口不外扩 + 盖住完整动作。

切段规则清单（GEPA 规则 prompt）

直觉：不是新模型；是 Macrodata 搜索得到的一组英文硬规则，贴进分段请求。

实现：completed_events_duration_prior_v1；样例 Step 02。消融表：整集 + 切段规则清单 → 0.1369。

规则 merge（三种）

直觉：在已有预测上用脚本合并相邻段，时间轴更好看，但 F1 全掉——不要默认开。

实现：完全相同文案 / 同动词物体 / 短间隙桥接；均为规则后处理，不再调 LLM。

C. 标注 / E2E 术语（含「为何 neighbor 掉分」）

raw relabel（397B 单路 raw 重标）

直觉：分段边界已经锁死，只换「谁来写句子」：用 397B 看该段若干 raw 帧重写 subtask。

实现：同一批 S2 边界（308 pred）；每段约 5 帧、最长边约 1120px。E2E 0.1388（语义 match 从自标的 41 提到 54）。

ffmpeg raw relabel

直觉：边界相同，只换视频解码/抽帧路径，检验标注是否对读帧实现过敏。

结果：E2E 0.1414，略好于默认 raw，常作为 selector 候选源之一。

neighbor relabel（邻段重标）

直觉：标当前段时，把上一 / 当前 / 下一段各抽少量帧一起给模型，希望「有上下文更准」。在 Qwen 上经常变成邻段上下文干扰：模型把邻段动作写进当前句。

实现：PREV/CUR/NEXT ≤5 帧，可带 seed prior（小模型或 397B raw 先验）。

为何 27B-prior neighbor → 0.108：分段仍锁在 0.2031，IoU match 仍是 79；语义 match 掉回约 42（接近自标）。 所以条形图上「突然降分」不是分段坏了，而是标注路径伤了句子。raw397-prior neighbor 可到 0.144，仍低于 selector。

candidate selector

直觉：每段先生成多条候选（raw / rawprior / seed / ffmpeg 等），再让 397B 选一句最像「完成操作」的短指令。

结果：E2E 0.1517（语义 match 59）。生产若要省调用，可同边界改用单路 raw（0.1388）。
